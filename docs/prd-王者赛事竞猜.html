01 项目概述

Project Overview
产品背景、目标定位、用户画像与产品范围

1.1 项目背景

《王者荣耀》作为国内最热门的移动电竞游戏，KPL（王者荣耀职业联赛）拥有庞大的观众群体。竞猜是电竞赛事生态中重要的互动形式，能够显著提升观众参与度和赛事黏性。

本项目旨在构建一个轻量级的内部赛事竞猜平台，为团队提供一个便捷的竞猜工具，用户通过虚拟金币对 KPL 赛事进行投注，体验竞猜乐趣。

1.2 产品目标

  • 核心目标：提供一个稳定、易用的赛事竞猜平台，支持赛季全生命周期管理
  • 用户体验：简洁直观的操作界面，关键路径（投注、查看结果）不超过 3 步
  • 管理效率：管理员可一站式完成赛季创建、赛程生成、赛果录入等操作
  • 内部使用：无需对外推广，聚焦功能完整性和稳定性

1.3 用户画像

用户类型角色核心需求使用频次
普通玩家BETTOR 每日签到、查看比赛、投注、查看结果和排行榜 每日 1-3 次
管理员ADMIN 赛季管理、战队管理、赛程生成、赛果录入、用户管理 赛季期间按需操作

1.4 产品范围

范围包含
功能范围 用户认证、赛季管理、战队管理、比赛管理、赛程生成、投注（单场+冠军）、动态赔率、自动结算、弃赛退款、排行榜、签到、管理后台
非功能范围 响应式 Web 界面、JWT 认证、数据持久化、CI 门禁
明确不包含 移动端 App、社交分享、支付系统、实时直播集成、多语言
该平台为内部工具，不涉及真实货币交易，所有金币为虚拟积分。

       ▼

       │
       ▼

3.3 结算流程

管理员录入比赛结果（设置胜者）
       │
       ▼
┌───────────────────────────────────┐
│        Settlement Engine          │
│  (Prisma 事务保证一致性)          │
├───────────────────────────────────┤
│  1. 查询该比赛所有 PENDING 投注    │
│  2. 遍历每笔投注：                  │
│     ├─ 猜中 → 状态 = WON           │
│     │         派发 amount × odds   │
│     ├─ 猜错 → 状态 = LOST           │
│     └─ 弃赛 → 状态 = REFUNDED       │
│            全额退款                 │
│  3. 更新战队胜负场统计              │
│  4. 淘汰赛：自动传播胜者至下一轮     │
└───────────────────────────────────┘
       │
       ▼
确认结算完成

  Match 3 ────────→  Winner 3               │
  Match 4 ────────→           ────────→           ────────→  Champion

5.2 核心业务规则

用户 & 金币系统

规则说明
初始金币新注册用户获得 100 金币
签到奖励每日签到领取 5 金币，截止日期为次日 00:00
封禁处理封禁用户无法投注、签到，已有投注正常结算
金币下限金币余额不能为负，投注金额必须 ≤ 当前余额
交易流水每一笔金币变动均记录 CoinTransaction，含 before/after 快照

投注规则

规则说明
投注条件比赛状态为 UPCOMING 或 LIVE 方可投注
唯一性约束同一用户对同一比赛只能投注一次（userId + matchId 唯一）
最低投注额每次投注至少 1 金币
最高投注额不设上限，受用户余额约束
赔率锁定投注时锁定当前赔率，后续赔率变化不影响已下注
冠军投注同一用户对同一赛季的同一战队只能投注一次（userId + seasonId + teamId 唯一）
冠军结算采用资金池模型，猜中者按投注额比例瓜分总投注池

赔率计算规则

adjustedPool = betTotalA + betTotalB + BASE_BET × 2
oddsA = clamp(adjustedPool / (betTotalA + BASE_BET) × PLATFORM_FEE, MIN_ODDS, MAX_ODDS)
oddsB = clamp(adjustedPool / (betTotalB + BASE_BET) × PLATFORM_FEE, MIN_ODDS, MAX_ODDS)

参数：
  BASE_BET     = 300（虚拟基础投注，防止极端赔率）
  MIN_ODDS     = 1.3
  MAX_ODDS     = 5.0
  PLATFORM_FEE = 1.0（无抽水）
  DEFAULT_ODDS = 2.0（初始默认值）
规则说明
动态计算每次新投注后自动重新计算对阵双方赔率
BASE_BET300 虚拟投注保底，确保赔率不会出现极端值
管理员覆写管理员可手动设置固定赔率，覆盖动态计算值
范围限制赔率始终在 [1.3, 5.0] 区间内

结算规则

规则说明
结算触发管理员录入比赛胜者后自动触发
事务保证整批结算在 Prisma 事务中执行，保证数据一致性
幂等性重复调用结算不产生重复派发（已结算投注跳过）
赔付计算WON = 投注额 × 投注时赔率；LOST = 0；REFUNDED = 全额退还
弃赛退款标记 FORFEITED 后，该比赛所有投注全额退款
晋级传播淘汰赛结束后，胜者自动填充到下一轮对应位置

赛程生成规则

规则说明
小组赛圆桌旋转算法（Circle Method），每个小组内所有队伍两两对决
淘汰赛支持两种模式：A1vsB2（交叉对阵）或按小组排名对阵
分级赛程小组赛循环赛 → 淘汰赛（Bo1），自动生成后续轮次占位
时间间隔每个比赛日之间的间隔由管理员在 UI 中配置

07 技术方案

Technical Architecture
系统架构、技术选型与接口设计

7.1 系统架构

┌─────────────────────────────────────────────────────────────────┐
│                    客户端 (Browser)                               │
│  React 18 + MUI 6 + Tailwind + Zustand + Axios                  │
└────────────────────────┬────────────────────────────────────────┘
                         │ HTTP / JSON
                         ▼
┌─────────────────────────────────────────────────────────────────┐
│                    服务端 (Express.js)                            │
├─────────────────────────────────────────────────────────────────┤
│  Routes (薄层) → Controllers (校验+格式化) → Services (业务)    │
├─────────────────────────────────────────────────────────────────┤

                         ▼

└─────────────────────────────────────────────────────────────────┘

7.2 技术选型明细

层级技术版本选型理由
前端 React^18.3成熟的前端框架，生态丰富
Material UI (MUI)^6.1开箱即用的组件库，快速搭建后台 UI
Tailwind CSS^3.4原子化 CSS，配合 MUI 实现灵活样式
Zustand^5.0轻量级状态管理，比 Redux 更简洁
React Router^7.0标准路由方案 + 权限守卫
后端 Express.js^4.21Node.js 最成熟的 Web 框架
Prisma^6.0类型安全的 ORM，自动生成 TypeScript 类型
JWT (jsonwebtoken)^9.0无状态认证，7 天过期
Zod^3.24请求校验，类型安全的 Schema 验证
数据库 SQLite—开发环境，零配置，文件级数据库
MySQL—生产环境，成熟的关系型数据库

7.3 架构分层原则

三层分离架构

层职责禁止
Routes 定义路由映射、注册中间件、参数透传 不包含任何业务逻辑或数据校验
Controllers 请求参数提取与校验（Zod）、响应格式化 不写业务逻辑，不直接操作数据库
Services 全部业务逻辑：校验、计算、数据库操作、事务 不直接处理 HTTP 请求/响应

08 非功能需求

8.1 性能要求

指标目标值说明

API 响应时间< 500ms95% 百分位响应时间
并发用户数≥ 50同时在线用户数支持

8.2 安全要求

类别要求
密码存储bcrypt 哈希存储，不存明文
认证方式JWT 令牌，7 天过期，Bearer 头传递
权限控制JWT 中间件 + Admin 中间件双层校验
输入校验Zod Schema 验证所有请求参数
文件上传multer 中间件，限制文件类型和大小
数据一致Prisma 事务保证投注/结算的 ACID

8.3 兼容性

类别支持范围
浏览器Chrome ≥ 90, Firefox ≥ 90, Safari ≥ 15, Edge ≥ 90
屏幕尺寸320px 起响应式适配，支持 PC/平板/手机
后端环境Node.js ≥ 18

8.4 可扩展性

  • 数据库切换：Prisma ORM 支持一键切换 SQLite ↔ MySQL，修改 provider + DATABASE_URL 即可
  • 模块化服务：12 个 Service 相互独立，新增功能只需新增 Service + Route + Controller
  • 前端组件化：5 大类 30+ 组件，页面由组件拼接，新增页面成本低
  • API 版本控制：路由以 /api/ 前缀统一管理，未来可通过 /api/v2/ 扩展

8.5 质量保障

维度措施
单元测试Vitest + 独立 SQLite 测试库，覆盖 4 个核心 Service（31 个测试用例）
类型检查TypeScript strict 模式，tsc --noEmit 前置检查
CI 门禁GitHub Actions：push/PR 自动触发 tsc → test 流水线
测试纪律集成测试禁止 mock 数据库，共享 Prisma 实例避免重复连接
错误处理全局 Error Handler 中间件，统一错误响应格式
